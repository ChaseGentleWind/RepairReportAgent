# 报修图片智能分析系统 - Agent 方案设计

## 一、系统架构

### 整体流程
```
用户上传图片 → FastAPI 接收 → 图片预处理 → LLM 分析 → 结构化输出
```

### 核心分层
1. **API 层** (FastAPI): 接收请求、参数验证、响应封装
2. **图片处理层** (Pillow): 格式转换、压缩、Base64 编码
3. **Agent 层** (LLM): 视觉推理、意图识别、结构化输出
4. **数据验证层** (Pydantic): 字段校验、类型约束

---

## 二、核心策略

### 1. 图片预处理策略
**目标**: 降低 API 成本，提升推理速度

- **格式统一**: 所有图片转为 RGB JPEG
- **尺寸压缩**: 长边限制 1024px，保持清晰度
- **编码标准**: Base64 Data URI 格式传输

### 2. 提示词设计策略
**核心思想**: Chain of Thought (思维链) 推理

#### 强制推理步骤
```
1. 空间扫描 → 确定视角（仰视/平视/俯视）
2. 物件识别 → 具体命名（如"空调出风口盖板"而非"空调"）
3. 状态演绎 → 4 条推理分支
4. 意图推导 → 用户真实需求
```

#### 4 条推理分支
- **分支 A**: 有明显破损 → 物理破损与脱落
- **分支 B**: 视觉异常但非破损 → 功能失效与环境异常
- **分支 C**: 手持物品/未安装 → 安装/加固/拆卸需求
- **分支 D**: 看起来完好 → **盲猜潜在故障**（核心创新）

**分支 D 示例**:
- 门的特写 → 推测：门关不上、锁打不开
- 空调完好 → 推测：异响、需要关闭出风口
- 碎纸机完好 → 推测：卡纸、不通电

### 3. 分类体系
5 个标准分类（Literal 枚举）:
1. **物理破损与脱落** - 裂开、松动、脱落
2. **机械卡阻与五金故障** - 门关不上、锁打不开、窗帘卡住
3. **功能失效与环境异常** - 异响、白屏、不亮、漏水
4. **安装/加固/拆卸需求** - 挂画、装挡风板、加固
5. **未知/需人工确认** - 极度模糊

### 4. 输出结构设计
**关键字段**:
```python
{
    "is_valid_image": bool,  # 图片有效性
    "spatial_and_state_analysis": str,  # 空间分析（新增）
    "reasoning": str,  # 推理过程
    "object_name": str,  # 具体物件名
    "issue_description": str,  # 问题描述
    "category": Literal[5种],  # 标准分类
    "confidence": Literal["High", "Medium", "Low"],
    "urgency": Literal["High", "Medium", "Low"]
}
```

**字段顺序策略**: 先推理 → 后结论（引导 LLM 思考）

## 三、技术方案

### 1. LLM 调用配置
```python
model: "qwen3.5-omni-flash"  # 通义千问多模态
temperature: 0.1  # 低温度保证稳定性
response_format: {"type": "json_object"}  # 强制 JSON 输出
max_tokens: 1000
```

### 2. JSON 解析策略
**问题**: LLM 可能在 JSON 后添加 markdown 标记

**解决方案**: 括号匹配算法
```
1. 查找第一个 '{'
2. 用栈匹配括号
3. 提取完整 JSON 对象
4. json.loads() 解析
```

### 3. 错误处理机制
- **重试策略**: 最多 3 次重试
- **异常分类**:
  - JSON 解析错误 → 重试
  - Pydantic 验证错误 → 记录并抛出
  - API 调用失败 → 重试

### 4. 性能优化
- **单例模式**: Agent 实例复用
- **异步处理**: FastAPI 异步端点
- **图片压缩**: 减少传输时间

## 四、关键创新点

### 1. 分支 D "盲猜"策略
**痛点**: 很多故障在图片上看不见（异响、卡住、失灵）

**解决**: 基于办公场景常识的潜规则推测
- 门的特写 → 通常是关不上
- 完好的电器 → 通常是功能故障

### 2. 空间分析前置
**作用**: 强制 LLM 先观察再推理
- 确定拍摄角度
- 寻找微小线索（缝隙、水渍、脱落边缘）

### 3. 具体化物件命名
**要求**: "空调出风口盖板" 而非 "空调"
**好处**: 提升维修工单的可操作性

## 五、数据流概览

```
1. 用户上传 (UploadFile)
   ↓
2. 图片预处理 (PIL + Base64)
   ↓
3. 构造消息 (System Prompt + Image + User Text)
   ↓
4. LLM 推理 (通义千问 API)
   ↓
5. JSON 解析 (括号匹配)
   ↓
6. Pydantic 验证 (字段校验)
   ↓
7. 返回结构化响应
```

## 七、总结

### 核心优势
1. **思维链推理**: 强制 LLM 按步骤思考，提升准确性
2. **盲猜策略**: 解决"看不见的故障"问题
3. **结构化输出**: Pydantic 严格验证，保证数据质量
4. **容错机制**: 重试 + 防御性解析，提升稳定性

### 适用场景
- 企业办公设施报修
- 需要快速分类和优先级判断
- 图片质量参差不齐的场景

### 技术栈
FastAPI + Pydantic + Pillow + 通义千问多模态 API
