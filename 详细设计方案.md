2.1 第一步：打标模型的选择

  参考方案建议用 Qwen3-VL-4B 或 Qwen2.5-VL 做快速打标。这里有个实际问题：

  - 当前项目调用的是阿里云 DashScope API，不是本地部署模型
  - DashScope 上没有 Qwen3-VL-4B 这个独立的轻量端点，有的是 qwen-vl-plus（较便宜）和
  qwen-vl-max（较贵）
  - 更务实的做法：用同一个 qwen3.5-omni-flash 模型，但换一个极简 prompt，只要求输出 3-5 个关键词的
   JSON，token 消耗极少，速度快

  这样可以避免引入第二个模型依赖，整个系统仍然只有一个 API Key、一个 base_url。

  2.2 第二步：向量库的选型细化

  参考方案提到 Milvus/Qdrant/FAISS，但没有给出选型依据。结合项目实际规模：

  ┌──────────┬──────────────────────────────────┬─────────────────────────────────────────────┐
  │   方案   │             适用场景             │               当前项目适合度                │
  ├──────────┼──────────────────────────────────┼─────────────────────────────────────────────┤
  │ FAISS    │ 纯 Python，无服务，适合 <10 万条 │ ✅ 最简单，直接 pip install                 │
  ├──────────┼──────────────────────────────────┼─────────────────────────────────────────────┤
  │ ChromaDB │ 有持久化，API 友好，适合中小规模 │ ✅ 推荐，有 LangChain 原生集成              │
  ├──────────┼──────────────────────────────────┼─────────────────────────────────────────────┤
  │ Qdrant   │ 需要 Docker，适合生产级          │ 项目已有 Docker，可作为升级路径             │
  ├──────────┼──────────────────────────────────┼─────────────────────────────────────────────┤
  │ Milvus   │ 分布式，适合百万级               │ ❌ 过度设计，当前 40-60 个 chunk 完全不需要 │
  └──────────┴──────────────────────────────────┴─────────────────────────────────────────────┘

  推荐 ChromaDB：项目已有 LangChain，langchain-chroma 直接集成，持久化到本地目录，Docker
  部署时挂载 volume 即可。

  2.3 第二步：Embedding 模型的选择

  参考方案推荐 bge-large-zh，这是个好选择，但需要本地运行（约 1.3GB 模型文件）。有两个替代路径：

  - 路径 A（本地 bge-large-zh）：精度最好，但需要额外安装 sentence-transformers，Docker
  镜像会增大约 2GB
  - 路径 B（DashScope Embedding API）：text-embedding-v3 模型，同一个 API
  Key，无需本地模型，但每次检索有网络延迟

  考虑到这批 PDF 只有 40-60 个 chunk，建议路径
  A：索引构建是一次性离线操作，运行时检索用本地向量，速度更快且无额外 API 费用。

  三、PDF 知识库构建的精细化方案

  3.1 各 PDF 的处理方式分类

  B 类：流程提取（提取派工路由信息）

  这类文档不直接描述设施状态，而是定义了维修流程和职责边界：

  C 类：跳过（不入库）

  MT-13 第 2 页是流程图，pdfplumber
  提取出来是乱序文字。处理方式：跳过流程图页，只提取文字描述页。MT-13 的核心信息在第 1 页和第 3
  页，已经足够。

  两阶段 Chain 的逻辑：

  阶段一（打标）：
    image → tagging_prompt | llm | keyword_parser
    输出：["地下车库", "倒车杆", "变形"]

  阶段二（检索 + 推理）：
    keywords → retriever.search() → sop_context
    sop_context + routing.get_dispatch(category) → dispatch_info
    image + sop_context + dispatch_info → repair_prompt | llm | parser
    输出：suggested_options + dispatch_info

  五、关键实现细节补充

  5.1 检索 query 的构造

  参考方案用关键词列表直接检索，但向量检索对自然语言句子效果更好。建议把关键词拼成一句话再检索：

  # 不好：直接用列表
  query = str(keywords)  # ["地下车库", "倒车杆", "变形"]

  # 好：拼成自然语言
  query = f"{keywords[0]}中的{keywords[1]}出现{keywords[2]}问题"
  # → "地下车库中的倒车杆出现变形问题"

  5.2 检索结果的过滤

  召回 top-3 后，需要设置相似度阈值过滤：

  - 相似度 > 0.75：注入 SOP 上下文，走 RAG 增强路径
  - 相似度 0.5-0.75：注入但标注"参考置信度较低"
  - 相似度 < 0.5：不注入，退回到当前的纯视觉推理路径（保底）

  这个降级机制很重要，确保 RAG 检索失败时系统仍然可用。

  5.3 Context 长度管理

  当前 prompt 已经包含：系统指令 + 故障词典 + 5 个 few-shot 示例，token 量已经不小。加入 SOP
  上下文后需要做取舍：

  - 精简 fault_dictionary.py：RAG 上线后，故障词典的作用被 SOP 检索部分替代，可以缩减到 2-3
  行简要说明
  - SOP 上下文限制：每个 chunk 约 150-200 字，top-3 召回约 450-600 字，控制在合理范围内
  - few-shot 示例动态化（后期优化）：从历史工单 xlsx_records.json 中动态检索最相关的 1-2
  个示例，替代现在硬编码的 5 个

  5.4 索引构建的触发时机

  indexer.py 应该是一个离线脚本，不在 FastAPI 启动时运行。建议：

  # 首次部署时手动运行
  python -m app.rag.indexer --rebuild

  # 或在 docker-compose 中作为 init container

  向量库持久化到 data/chroma_db/ 目录，Docker 挂载为 volume，不随容器重建而丢失。
